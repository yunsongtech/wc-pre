# 《世界杯智能预测与战术观赛系统》项目需求文档 (PRD v2.0)

## 1. 项目概述 (Project Overview)

* **项目名称：** WorldCup Oracle & Wake (暂定名)
* **产品定位：** 一款将“大数据比分预测”与“智能作息管理”深度融合的创新赛事平台。它不仅为用户提供科学的比分概率参考，更作为“懂球的智能看球搭子”，通过动态算法自适应接管用户的观赛闹钟与全屋智能设备，解决熬夜球迷“睡眠碎片化”与“错过名场面”的终极痛点。

## 2. 目标用户与痛点分析 (Target Audience & Pain Points)

* **硬核熬夜球迷（白天需上班）：** 想看直播但精力有限。**痛点：** 订了闹钟爬不起来，或者起来了发现是一场毫无尿点的 0-0 闷战，严重浪费睡眠。
* **碎片化观赛球迷：** 精力极度有限，只想看最精彩的段落。**痛点：** 醒来时比赛已经结束，只能看无感的集锦，错过了实时见证点球大战或绝杀的历史瞬间。
* **数智生活爱好者：** 追求科技感。**痛点：** 传统的体育 App 和智能家居脱节，观赛缺乏沉浸式氛围。

## 3. 核心功能需求 (Core Functional Requirements)

### 3.1 数据接入与处理引擎 (Data Engine)

* **MVP 零成本数据源（已实现）：**
  * **FM CSV：** 球队六维属性、FIFA 排名、阵型（`data/teams.csv` + 列映射配置）
  * **World Bank API v2：** 国家队所属国 GDP、人口（宏观实力雷达维）
  * **Wikimedia REST API：** 队徽缩略图 URL、百科摘要
* **ELO 基准：** 从 FM CSV 的 `fifaRank` 反推：`ELO = 2100 - 25 × (rank - 1)`
* **后续扩展（未实现）：** 纳米数据 WebSocket 毫秒级实时流、Kafka/Flink 流计算
* **即时情报：** 开赛前 45 分钟的官方首发名单、阵型、伤病更新（Phase 2+ 接外部 API）
* **实时数据流：** 当前 Phase 5 使用 Go mock WebSocket；生产环境目标 < 3s 端到端延迟

### 3.2 智能预测与指数核心 (Prediction Core)

在传统的比分预测基础上，新增两组面向“闹钟模块”的算法模型：

* **胜平负/比分预测：** Go Engine 双变量泊松模型；Semantic 层（DeepSeek）输出 `lineupTheta` 修正 μ/ν
* **「赛事精彩指数」(Excitement Index)：** Engine 计算（0-10）；DeepSeek 输出 `excitementAdj` 微调
* **「实时动态威胁度」( Live Threat Score)：** Go LTS 纯函数 + mock SSE/WebSocket 推送（目标阈值 0.82）

### 3.3 核心亮点：智能战术闹钟模块 (Tactical Alarm Module) —— *NEW*

#### 3.3.1 「AI战术唤醒」 (Tactical Wake-up)

* **业务逻辑：** 闹钟不再是单一的铃声，而是结合赛前 45 分钟首发情报的“战术早报”。
* **产品行为：**
* 赛前 15 分钟触发闹钟。
* **音频/视觉播报：** 闹钟响起时，AI 语音直接播报预测简报（例：“*起床啦！距离法阿大战还有15分钟。突发情报：梅西确认首发，阿根廷阵型变为 4-3-3 强攻。系统最新调高阿根廷胜率至 54%，预计最可能比分 2-1，这场比赛精彩指数高达 9.2，不容错过！*”）。



#### 3.3.2 「硬核免打扰」 (Predictive Sleep Guard)

* **业务逻辑：** 用算法对比赛价值进行前置评估，保护用户睡眠，防止“无效熬夜”。
* **产品行为：**
* **闷战劝退：** 若系统计算出某场凌晨 3 点的比赛「精彩指数」极低（如两支打平即可携手出线的防守型球队，0-0 概率 > 60%），系统将在前一天 22:00 弹窗提示：“*算法检测到明晨比赛沉闷概率极高，已自动为您建议‘睡眠保护模式’。是否关闭闹钟？明早 7:00 我们将为您推送‘无剧透3分钟高光集锦’。*”
* **防爆肝提示：** 当检测到用户已连续 3 天熬夜，且当日比赛非生死战时，主动调低闹钟唤醒级别或劝导睡眠。



#### 3.3.3 「高能节点触发器」 (High-Light Trigger)

* **业务逻辑：** 针对不想看全场、只想看“名场面”的碎片化用户，由算法在比赛进行中**实时中途唤醒**。
* **产品行为：**
* 用户选择“高能唤醒模式”后，开赛时不设闹钟，用户安心睡觉。
* **触发条件：** 当比赛进入下半场且满足以下任一条件时，瞬间引爆强力闹钟：
1. 裁判判罚**点球**或正在进行 **VAR 关键红牌审查**。
2. 比赛第 80 分钟，双方比分依然是**焦灼的平局**或 **1 球差距**（绝杀高发期）。
3. 淘汰赛正式进入**加时赛或点球大战**。





### 3.4 场景化交互与智能家居联动 (UI/UX & IoT Linkage)

* **App 观赛大厅：** 比赛列表不仅显示时间，还直观展示「战术闹钟状态」图标（已开启全场唤醒 / 已开启高能触发 / 睡眠保护中）。
* **IoT 全屋观赛接力（高级扩展）：**
* **微弱前置：** 闹钟响前 10 分钟，卧室灯光模拟日出缓慢亮起，音箱播放微弱的球场环境噪音，让用户自然浅睡。
* **电视联动：** 闹钟确认解除（用户点击起床）后，客厅电视/投影仪自动开机并切入对应的直播频道。
* **进球全屋狂欢：** 预测系统检测到进球，立刻联动智能灯带闪烁主队国旗色，音箱模拟全场欢呼。

## 4. 非功能性需求 (Non-Functional Requirements)

| 需求维度 | 指标与验收标准 | 对应核心功能 |
| --- | --- | --- |
| **实时性 (Latency)** | 赛场事件数据到手机闹钟触发的端到端延迟必须 **< 3秒**。 | 「高能节点触发器」（必须保证用户醒来时点球还没罚出去） |
| **推送到达率 (Push Delivery)** | 高能短信/系统级 Push 到达率必须 **> 99.9%**，需接入头部厂商的系统级推送通道（华为/小米/Apple APNs）。 | 「高能节点触发器」 |
| **稳定性 (Reliability)** | 凌晨 2:00 - 5:00 期间，战术早报 AI 语音合成服务可用性需达到 **99.95%**。 | 「AI战术唤醒」 |
| **功耗优化 (Battery)** | App 在后台常驻监听高能推送时，每小时耗电量需 **< 0.5%**，避免因监听导致手机断电关机。 | 所有闹钟功能 |

## 5. 项目路线图 (MVP 规划建议)

* **Phase 1 (基础版 - 当前)：** Adapter + Engine + Semantic + 雷达图 UI；`/api/v1` REST；Docker 部署 Go 后端
* **Phase 2 (智能作息版)：** 真实 Nano Data WebSocket + 系统级 Push + 阿里云语音外呼
* **Phase 3 (全屋生态版)：** Home Assistant / 米家真实 Webhook 联动（当前为 mock + env 开关）
